# Правила работы Codex с Git

Главное правило:
Codex не должен создавать PR с конфликтами. Если возникает конфликт, Codex не должен просить пользователя решать его вручную. Codex должен сам пересобрать свою ветку заново от свежей основной ветки и повторно применить изменения.

## Основной workflow

Перед началом любой задачи:

1. Определи основную ветку репозитория:
   - main
   - master
   - или default branch из origin.

2. Обнови репозиторий:

   git fetch --all --prune

3. Никогда не начинай новую задачу от старой рабочей ветки.

4. Создай новую ветку строго от свежей основной ветки:

   если основная ветка main:

   git switch -c codex/<short-task-name> origin/main

   если основная ветка master:

   git switch -c codex/<short-task-name> origin/master

5. Вноси изменения только в этой новой ветке.

## Перед commit и PR

Перед созданием commit / PR обязательно:

1. Сохрани список изменённых файлов и суть изменений.

2. Обнови origin:

   git fetch origin

3. Выполни rebase на актуальную основную ветку:

   если основная ветка main:

   git rebase origin/main

   если основная ветка master:

   git rebase origin/master

4. Если rebase прошёл без конфликтов:
   - проверь отсутствие конфликтных маркеров;
   - проверь diff;
   - запусти доступные проверки;
   - только после этого создавай commit и PR.

5. Если rebase дал конфликт:
   - НЕ создавай PR;
   - НЕ коммить конфликтные файлы;
   - НЕ оставляй маркеры конфликта;
   - НЕ проси пользователя чинить конфликт вручную.

Вместо этого сделай так:

   git rebase --abort

Затем:
   - сохрани свои изменения как patch, если это возможно;
   - создай новую ветку от свежей origin/main или origin/master;
   - заново примени нужные изменения поверх свежего кода;
   - не восстанавливай старую версию файла целиком, если в main уже есть новые изменения;
   - если файл нужно заменить полностью, сначала возьми его актуальную версию из origin/main или origin/master, потом перепиши только нужную реализацию;
   - повтори проверки.

## Жёсткое правило при конфликте

Если конфликт возник, Codex должен считать текущую рабочую ветку устаревшей.

Правильное действие:
- не чинить конфликт поверх старой ветки;
- не мержить с конфликтами;
- не создавать PR;
- пересобрать изменение заново на свежей основной ветке.

## Запреты

Запрещено:

- создавать PR, если GitHub показывает conflicts;
- пушить ветку с незавершённым rebase;
- коммитить файлы с маркерами:

  <<<<<<<
  =======
  >>>>>>>

- делать force push в main/master;
- перетирать актуальную main/master старой версией файла;
- запускать автоформатирование по всему проекту без необходимости;
- трогать файлы, которые не относятся к задаче;
- держать несколько параллельных PR, которые меняют один и тот же файл.

## Если нужно перетереть старую реализацию

Если задача требует заменить старую реализацию:

1. Сначала обновись от свежей origin/main или origin/master.
2. Возьми актуальную версию файла.
3. Замени только нужный модуль, функцию или блок.
4. Не откатывай изменения, которые уже есть в основной ветке.
5. После замены снова сделай:

   git fetch origin
   git rebase origin/main

   или

   git fetch origin
   git rebase origin/master

6. Если снова появился конфликт — снова пересобери ветку заново от свежей основной ветки.

## Проверки перед PR

Перед PR обязательно выполнить:

   git status --short --branch

   git diff --check

   git grep -n "<<<<<<<\|=======\|>>>>>>>" -- . || true

Если команда git grep нашла конфликтные маркеры — PR создавать запрещено.

Если задача затрагивает Python/Streamlit-код, нужно проверить хотя бы:
- импорт основных модулей;
- запуск приложения, если возможно;
- отсутствие синтаксических ошибок.

## Финальный ответ Codex

В конце каждой задачи Codex обязан написать:

1. От какой ветки создана рабочая ветка.
2. Выполнялся ли git fetch.
3. Выполнялся ли rebase на origin/main или origin/master.
4. Были ли конфликты.
5. Если конфликты были — была ли ветка пересобрана заново.
6. Проверялось ли отсутствие конфликтных маркеров.
7. Какие файлы изменены.
8. Какие проверки запущены.
9. Готов ли PR к merge без конфликтов.
